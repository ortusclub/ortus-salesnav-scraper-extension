Sales Nav Scraper · v3.10 redesign review

Make it obvious.
Then make it safe.

A side-by-side review of the proposed v3.10 changes. Three problem areas: Single Link is unreachable for users not already on a Sales Nav tab, Resume copy is abstract ("Resume" / "Start fresh" tells you nothing about what's at stake), and several controls feel anonymous — dropdowns where radio cards belong, ghost buttons where hierarchy belongs. Underneath, the same editorial palette and typography. Everything visible meets WCAG AA.

Touch targets
Every interactive control is at minimum 44×44 px. Many today are 32 px — too small for thumbs and trackpads.
Status never color-only
Pills and chips pair color with an icon and a text label so colorblind users get the same signal.
Concrete labels
Buttons describe what will happen, not what they're called. "Resume" → "Pick up at page 4 (73 leads safe)".
01 · Discoverability

Quick / Batch toggle, pinned to the masthead.

Today single-link mode is gated behind two conditions: (a) you must already be on a Sales Nav tab, and (b) you have to spot a tiny ghost button under the Connect CTA. If either fails, single mode is unreachable. The new toggle is always visible, always one click away.

02 · The worst dropdown

"Tab name per job" → three radio cards.

This is the dropdown that hides the most decision. Three modes (auto-number / fixed name / column lookup) each reveal a different sub-input. Hiding them inside a select means users can't compare the options — they have to click each one to discover what it changes. Radio cards put all three options on screen with examples, so the choice is visible.

Current

Proposal
Three things visible at once: the choice, the explanation, and a concrete example for each. Sub-inputs (column picker for option 3) appear inline below the selected card — never hidden behind a click.
03 · The column-pickers

Source tab and URL column — surface the data, not the spreadsheet jargon.

Selects are unavoidable when the option count is 50+ (every column letter). But today's selects show only the column label ("D") with no preview. The new pattern shows the column letter and the column header text and a preview of the first cell, so users pick by content rather than letter.

04 · After an interruption

Concrete state in the button label, not abstract verbs.

"Resume" tells you nothing. Resume what, from where, with what already saved? "Start fresh" sounds friendly — too friendly for an action that throws away your previous work. The proposal puts the saved state in the primary button's label and demotes the destructive option to a quiet text link with a confirmation step.

05 · Mid-run controls

Pause / Recover / Stop → grouped, hierarchized, plain-language.

Three ghost buttons in a row gives no hint of which is safe and which is destructive. "Recover" is engineering jargon — users don't know when to press it. "Stop" looks identical to "Pause" but throws away the run. The proposal groups them, adds an explainer for Recover, and gates Stop behind a confirmation.

06 · Accessibility, made visible

The commitments are in the design, not just in the spec.

These aren't abstract checklists — every item below is something you can see in the new design. WCAG AA on every color combo. 44 px minimum for any tappable thing. Status communicated by icon + text + color (never just color). Real focus rings. Reduced-motion respect.

Touch targets

Apple's HIG and Material both call for 44 px minimum on tappable elements. Today many of our pills are 32 px. The new buttons are 40–44 px and the dropdowns/cards are 44 px+.

Status — never color-only

A user with deuteranopia (red-green colorblindness) can't tell a green chip from an amber one. Every status pairs an icon and a text label with the color so the signal is multi-channel.

Real focus rings

Keyboard users currently get the browser default — sometimes nothing on dark backgrounds. The new design uses a 2 px gold-deep ring with 2 px offset, identical to the brand accent. Tab through to see it; the simulated state below shows what it looks like.

Reduced motion

All animation respects prefers-reduced-motion. The pulse on running indicators stops; the slide-in on alarms becomes an instant fade. No bouncing, no parallax, no spring physics.

"If the user said 'less motion,' the system says 'less motion.'"

Real labels

Every input has a real <label for=…>. Placeholders are examples, never used as labels — they vanish on focus and screen readers don't read them reliably. Status messages live in aria-live regions so updates are announced without yanking focus.

What this sketch does

One HTML file, no build step. Open it in any browser. Every visual primitive is the same as the real popup so the comparison is honest — same fonts, same colors, same spacing rhythm. The only differences are the changes being proposed.
